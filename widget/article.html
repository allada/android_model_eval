I Gave Frontier Models an Android Phone and a Pass/Fail Test Suite

Or: why I think phone-using LLMs need regression tests, not just demos.

Android · LLM · Benchmark · Replay

The moment this project became interesting was not when a model succeeded. It was when I watched one look straight at the airplane mode tile, explain that it was about to tap it, and then miss.

That was the moment this stopped feeling like prompt engineering and started feeling like systems engineering.

I did not want another vague answer to "how good are UI agents?" I wanted a narrower answer to a much more practical question:

If you give a model only the kinds of actions a human has on a phone—look at the screen, tap, swipe, long-press, press a physical button—can it reliably complete small real tasks? And can you score that without hand-wavy judgment calls?

So I built a small Android harness to test exactly that.

This is not a paper. It is not a universal leaderboard. I do not work on model training, and I am not claiming this is how everyone in the industry should already be doing things. This is just a concrete system I built, the methodology I used, and the observations that fell out of it.

The rule: no superpowers

The core constraint was simple: the model should interact with the phone the way a person would.

That means the agent side of the system only gets a very small action surface:

  • screenshots
  • taps
  • swipes
  • long-presses
  • a few physical button events
  • short waits for UI animations to settle

Under the hood, the bridge is backed by adb, but that is for setup, reset, and verification. The model itself does not get to cheat with shell access or private app APIs.

In the code, the MCP bridge exposes tools like get-device-session-info, screenshot, tap, swipe, long-press, key-event, and sleep. The hardware button tool is intentionally narrow. On the device side I kept it to physical buttons, not a magical "home" escape hatch. If the model wants to go home on a gesture-based Android device, it has to figure out the swipe.

That sounds like a small detail, but it matters. The whole point was to test interaction under something closer to human constraints, not "Android, but with hidden powers."

The harness is intentionally thin

The architecture ended up being simpler than people often assume these systems need to be:

  1. The harness creates a device session.
  2. It restores a clean baseline snapshot.
  3. It runs task-specific setup.
  4. It starts a screen recording.
  5. It hands the task prompt and session ID to the model.
  6. The model interacts through the MCP bridge.
  7. The harness stops recording and verifies the final device state with adb.
  8. The session is cleaned up.

That separation mattered.

The admin side exists to make tests deterministic: snapshot loading, setup commands, file installs, emulator console access, screen recording, and final verification.

The agent side exists to stay constrained: screenshots in, UI actions out.

I also wrote the device pool so sessions can be swapped with emulator snapshots. That let me build toward a larger parallel system even though this write-up is intentionally about a much smaller benchmark.

Why I cared so much about binary pass/fail

I did not want a fuzzy score. I wanted tests that end with a boring, deterministic answer.

  • Is airplane mode on? Check settings get global airplane_mode_on.
  • Was the alarm created? Check dumpsys alarm for a pending DeskClock alarm at 17:00.
  • Was Firefox Focus removed? Check pm list packages and make sure org.mozilla.focus is gone.

That is the design philosophy in one sentence:

If the alarm is not set, the task failed.

I am a big believer that tests should get more deterministic as you move down the stack, not less. For this kind of work, I would much rather have a small pile of clean pass/fail tasks than a larger pile of squishy ones with a lot of narrative scoring glued on afterward.

That also changes how I think about repeated runs. I am not re-running tests because the individual check is fuzzy. I am re-running them because the model is stochastic, and I want the pass rate of a clean test.

That is a very different mindset.

The tasks are small on purpose

The current stable task set is intentionally narrow:

  • turn airplane mode on
  • turn airplane mode off
  • set an alarm for 5:00 PM
  • uninstall Firefox Focus

There is also an SMS verification-code task in the codebase, but I am leaving it out of the main comparison until I tighten it up more. I would rather publish fewer tests I trust than more tests I have to explain away.

These tasks are not meant to be exhaustive. They are meant to be concrete.

Each one is small enough to reason about, hard enough to expose real failures, and verifiable without another model in the loop.

The uninstall test is a good example of why this gets interesting quickly. The harness installs Firefox Focus if it is missing, then asks the model to find and remove it. The prompt describes it as "the firefox logo, but purple." That means the model has to do more than OCR a label. It has to connect the request to an icon that is close semantically, not identical to the default mental image most people have of Firefox.

The alarm task is another favorite because it is just annoying enough to be real. It is multi-step, UI-heavy, and easy to verify afterward.

What I got back was not a leaderboard. It was a regression-testing story.

Across the four stable tasks, I ended up with 1,500 runs total across the model variants I tested. Aggregate pass rates by task looked like this:

Task Aggregate pass rate
Turn on airplane mode79%
Turn off airplane mode65%
Set alarm for 5:00 PM68%
Uninstall Firefox Focus62%

That is already enough to make one point clearly: these are not impossible tasks. They are the sort of basic phone operations a useful UI agent should eventually do almost all the time. Yet even strong models still miss them with surprising regularity.

The result I found most useful was not "which vendor won." It was that a newer model can still underperform an older one on concrete tasks.

In my Codex runs, gpt-5.3-codex beat gpt-5.4 on three of the four stable tasks:

Task gpt-5.3-codex gpt-5.4
Turn off airplane mode55%45%
Turn on airplane mode74%90%
Set alarm for 5:00 PM75%69%
Uninstall Firefox Focus74%67%

I care about that result for one reason: it is exactly the kind of thing regression tests are supposed to catch.

Without a task-level harness, that kind of movement is easy to miss. You might still have a model that feels better in some demos, or better on some internal evals, or better at tool calling in the abstract. But on a small real interaction task, it may have just gotten worse.

That does not mean I am claiming a grand conclusion about all model quality. It means I built a narrow system, ran it many times, and got a very practical reminder that progress is uneven.

One important caveat: I would treat within-family comparisons here as stronger than cross-family rankings. The overall harness is the same, but provider wrappers are not perfectly identical because the vendor CLIs expose different control surfaces. For me, the useful takeaway is not "vendor A beats vendor B forever." It is "small, repeatable UI tasks can reveal changes that matter."

The replay UI turned out to be as important as the score

I did not build the visualizer as eye candy. I built it because I do not trust black boxes I cannot inspect.

The widget replays each run with:

  • the recorded device video
  • a reconstructed "agent view" from the screenshots the model requested
  • the model's visible messages
  • token stats over time
  • overlays for taps, swipes, long-presses, screenshots, and key events

That matters because tests are not just for measurement. They are debugging tools.

A final pass rate can tell you that something regressed. A replay can tell you where to look.

That was useful in two different ways.

First, it helped me debug the harness itself. When you are building screenshot cropping, timing, recording, and coordinate-based actions across multiple systems, it is very easy to accidentally test the wrong thing. The visualizer was my sanity check that the system was actually showing the model what I thought it was showing.

Second, it made failures legible. Watching a model get lost in a UI is much more informative than staring at a CSV.

One debugging detour that felt worth keeping

I started with much larger screenshots.

That was expensive, awkward, and in at least one memorable case I watched a model recognize the correct target, say it would tap it, and then misclick badly enough to open the wrong settings pane.

Later, while trying to reduce token usage, I moved the system toward cropped and downscaled screenshots instead of handing the whole screen over every time. In the current code, the screenshot path supports region crops and scaling, which made it much easier to constrain what the model looked at and to reconstruct that view later in the widget.

I do not want to oversell that into a scientific claim about model vision internals. I did not design this project to isolate that effect cleanly. What I am comfortable saying is simpler:

A replayable harness makes it much easier to notice system-level details that are worth investigating.

That is the kind of observation I would turn into a follow-up ticket, not a headline.

What I would change in v2

If this were graduating from "small, opinionated engineering benchmark" into something more production-grade, I would push it in three directions.

1. Turn each task into a matrix, not a single scenario

Right now the stable tasks are intentionally controlled. The next step would be to add structured variation:

  • move icons around
  • vary app state
  • add distractors
  • include stale messages vs recent messages
  • test negative cases like "the app is not installed"
  • vary Android versions and device defaults

Same intent, many realizations. That is how you move from a good regression test to a more robust capability probe.

2. Keep a small number of expensive end-to-end tests, then factor the rest downward

A full emulator loop is high fidelity, but it is not the only level that matters.

Once you understand what a task is really measuring, many follow-up checks can be much cheaper:

  • screenshot-only tests
  • limited-action tests
  • cropped-region perception tests
  • targeted spatial-grounding checks

The full end-to-end runs should stay around as goldens. But they do not all need to be full end-to-end forever.

3. Keep the bar boring

I think it is easy to make these tests too clever. I would rather keep them boring and hard to fake.

Phone agents do not need superhuman puzzles right now. They need to stop missing the basics.

The main thing I took away

I do not think the lesson here is "phone agents are doomed." In fact, the opposite is true: some of these models are already good enough that the failures are more interesting than the successes.

The lesson I took away is much more boring, which is probably why I trust it:

If you want models to operate software, treat those behaviors like software capabilities. Give them pass/fail tests. Record the runs. Re-run them when the model or the wrapper changes.

That is not a grand theory of AI. It is just a practical way to stop lying to yourself.

And after building this, that is the conclusion I keep coming back to:

I do not need another benchmark full of abstract percentages. I need a growing pile of small, deterministic, replayable regression tests.
